# PB Ranker — Diagnosis & Flaw Inventory

> **Scope:** the personal-best (PB) detection → ranking → content-selection path.
> Covers both live rankers (`legacy/swim_content_v5/ranker.py` and `legacy/swim_content/ranker_v3.py`),
> the PB detectors (`recognition_swim/achievements/`, `swim_content_v5/achievements/pb.py`),
> the PB-discovery baseline path (`pb_discovery/`, `pipeline/pb_bridge.py`), and the docs that describe them.

**Method.** All PB-linked test files were run first — **363 tests pass** (180 PB-core + 183 ranker-adjacent), so every flaw
below is latent: real behaviour the current suite does not catch. An 8-dimension audit produced 84 raw findings, deduped to 62.
Each was put through an adversarial reproduce-vs-refute verification pass; the headline behavioural claims were additionally
re-checked by hand with direct probe scripts against the real code.

**Verdict legend.** `CONFIRMED` = both an independent reproducer *and* a hostile refuter agreed it is real (or a hand-probe
demonstrated it). `PLAUSIBLE` = one side confirmed, the other was inconclusive. `UNVERIFIED` = the automated verifier was
interrupted (session limit); still evidenced by the finder.

## Summary

**62 distinct flaws** — 3 critical · 20 high · 30 medium · 9 low.

| Severity | Count | What it means here |
| --- | --- | --- |
| 🔴 Critical | 3 | Factually wrong content ships (a fabricated or false PB reaches a card). |
| 🟠 High | 20 | Wrong ranking order, silent data loss, or a crash on realistic input. |
| 🟡 Medium | 30 | Drift, dead code, or coverage holes that mislead maintainers and mask the above. |
| ⚪ Low | 9 | Minor correctness / explainability / robustness nits. |

**By category:** Correctness 20 · Docs drift 12 · Dead / orphaned code 10 · Silent failure 5 · Determinism 4 · Integration 4 · Test gap 4 · Explainability 3

### The through-line

The PB path works on the happy case but has **no defensive floor**. Three themes recur:

1. **Fabricated baselines feed fabricated PBs.** The discovery parser will turn a relay split, or a calendar date, into a
   "previous best" time, and the detectors have no plausibility cap — so a −165s "new PB" can reach a card (F01, F02, F20, F25).
2. **The ranking tables have silently drifted from the achievement vocabulary.** Live achievement types
   (`official_pb_confirmed`, `biggest_drop_of_meet`, `relay_strong_performance`, `club_record`, meet levels `regional`/`international`)
   are missing from the ranker's lookup tables, so the strongest moments quietly get the default score and rank *below* weaker ones
   (F05, F07, F08, F12, F19).
3. **The documentation describes a ranker that does not exist.** `docs/RANKING.md`'s entire formula, weights file, tuning knobs and
   audit-trail field are fabricated, and the code paths they name are 0-byte stub files (F30, F38–F43, F48).

## 🔴 Critical

### F01 — Heuristic PB parser turns relay rows into individual-event baselines, shipping absurd false 'new PB' cards (e.g. -165.10s)

- **Where:** `src/mediahub/pb_discovery/parse_pbs.py`:139
- **Category:** Correctness  ·  **Verdict:** CONFIRMED
- **What's wrong:** _heuristic_extract_pbs builds the event as f"{dist_m.group(1)}m {stroke}" from any row containing a distance and a stroke word. A relay row like '4 x 100m Freestyle Relay 3:45.00' matches _DISTANCE_RE (100) and _detect_stroke ('free') and becomes PBRow(event='100m Freestyle', time='3:45.00'); '4 x 50m Medley Relay 1:52.33' becomes '50m Individual Medley'. pb_bridge then keys these as 100FRSC / 50IMSC baselines. Any individual swim in that event then 'beats' the relay time and PBConfirmedDetector fires.
- **Failure scenario:** A discovered profile/results page that lists the swimmer's relay legs (common on club and meet-results pages that pass the name+club identity gate) supplies the only entry for an event key; every individual swim in that event is announced as a huge new PB with a fabricated prior time on the generated card.
- **Evidence:** Probe output: event='100m Freestyle' time='3:45.00' from the relay row; bridged key='100FRSC': [225.0]; PBConfirmedDetector on a 59.90 swim produced: "Alex Doe sets new PB: 59.90 in 100m Freestyle (SC) (was 3:45.00, -165.10s)" and "Alex Doe sets new PB: 30.50 in 50m Individual Medley (SC) (was 1:52.33, -81.83s)" (probe_pb.py PROBES 5-7).
- **Refinement from verification:** The finding is fully accurate but understates scope in one respect: the defect is NOT confined to the heuristic fallback. The primary interpreter path (parse_pbs_from_page with use_interpreter=True, the production default) makes the same mistake — on the same relay-listing page it returned event='100m Freestyle' time='3:45.00' and event='50m Individual Medley' time='1:52.33' (source=interpreter) at confidence 0.9, higher than the heuristic's capped 0.6. So relay rows corrupt PB baselines on both extraction paths, and a fix limited to _heuristic_extract_pbs (e.g. rejecting rows matching /\b\d\s*[x×]\s*\d/ or containing 'relay') would leave the interpreter path (mediahub.interpreter events induction) still shipping the same false PBs.

### F02 — PB detectors ignore same-meet swims: a final slower than the swimmer's own heat still ships as 'sets new PB'

- **Where:** `legacy/swim_content_v5/achievements/pb.py`:71
- **Category:** Correctness  ·  **Verdict:** CONFIRMED
- **What's wrong:** PBConfirmedDetector.detect compares each swim only against history.best_time_in_event (the web-lookup baseline). The all_results parameter is accepted but never used, and report.py runs detectors per-swim with no folding of earlier same-meet swims into the baseline. So when a swimmer beats the online baseline twice in one meet (heats then final), BOTH swims fire pb_confirmed — including the slower one, which is factually not a PB. The safeguard that handles this (same-meet exclusion, same_meet_excluded_count) exists only in legacy/swim_content_pb/matcher.py, which the live V5 path never calls; pipeline_v4._v5_decision_from_achievement hardcodes same_meet_excluded_count=0.
- **Failure scenario:** Heats+finals meet: swimmer PBs in the heat (60.50, old PB 61.00), swims 60.90 in the final. Two 'new PB' cards are generated; the final's card claims a PB that is not one, with the wrong prior (1:01.00 instead of 1:00.50). pb_magnitude_* and multi_pb_weekend double-count the same event too.
- **Evidence:** Probe PROBE 8: baseline 61.0; heat 60.50 -> 'Alex Doe sets new PB: 1:00.50 ... (-0.50s)'; final 60.90 -> 'Alex Doe sets new PB: 1:00.90 in 100m Freestyle (LC) (was 1:01.00, -0.10s)' — a false PB claim, since the swimmer went 1:00.50 hours earlier at the same meet. PBImprovementMagnitudeDetector has the identical structure (lines 148-156).
- **Refinement from verification:** Minor precision: pb_magnitude_* double-fires only when BOTH same-meet drops exceed the 0.5% 'notable' threshold (in the probe scenario the final's 0.16% drop kept the magnitude detector silent); the structural defect in PBImprovementMagnitudeDetector (same static baseline, all_results unused, lines 148-156) is otherwise exactly as described. All other claims verified verbatim, including the pipeline_v4.py hardcoded same_meet_excluded_count=0 (lines 950 and 1066) and the multi_pb_weekend event-count inflation via report.py's no-dedup pb_by_swimmer append.

### F03 — Club records and milestones ship with post_angle="recap_mention" because the ranker's inline _POST_ANGLE_MAP was never updated for Phase W, and the builder's or-chain can't correct it

- **Where:** `legacy/swim_content_v5/ranker.py`:352
- **Category:** Correctness  ·  **Verdict:** CONFIRMED
- **What's wrong:** rank_achievements stamps post_angle from an inline _POST_ANGLE_MAP that lacks the Phase-W types (club_record, club_debut, race_milestone_25/50/100/250/500, first_event_swim), defaulting them to "recap_mention". The corrected mapping exists only in src/mediahub/content_pack/builder.py:_TYPE_TO_ANGLE (lines 48-56: club_record→"club_record", milestones→"milestone"), but _enrich_item (builder.py:93-94) computes item.get("post_angle") or a.get("post_angle") or _TYPE_TO_ANGLE.get(atype, ...) — the ranker already set the wrong truthy value on both the RankedAchievement and the Achievement, so the correct fallback never fires. post_angle is user-visible: it's humanised as the card's angle label (web/web.py:51929) and injected into the caption-LLM prompt as "Post angle: ..." (recognition/copy_text.py:89-91). Per ranker.py:71, club_record is "the highest-emotion club moment — outranks every PB and gold" (magnitude 1.1), yet it is presented and captioned as a generic recap mention. The PostAngle enum's CLUB_RECORD/MILESTONE members and their POST_ANGLE_LABELS ("New club record", "Club milestone") in recognition/schema.py are consequently unreachable via the ranker path.
- **Failure scenario:** A swimmer breaks a club record. The card ranks high (magnitude 1.1) but its angle label reads "Recap mention", and the caption prompt tells the LLM the post angle is a recap mention — the club's biggest moment ships with generic recap framing instead of the built-and-labelled "New club record" angle.
- **Evidence:** Executed probe: rank_achievements([Achievement(type="club_record", ...)], MeetContext(meet_level="county")) → priority 0.71, post_angle stamped "recap_mention" on both objects and in to_dict(); then _enrich_item(ra.to_dict()) → final post_angle "recap_mention". Same for race_milestone_100 and club_debut. _POST_ANGLE_MAP (ranker.py:352-379) has no club_record/club_debut/race_milestone_*/first_event_swim keys; _TYPE_TO_ANGLE (builder.py:21-57) has all of them under the comment "# Phase W — club history angles".
- **Refinement from verification:** All factual claims in the finding hold exactly as written. Severity is better characterised as high than critical: the ranking itself is unaffected (club_record still carries magnitude 1.1 and ranks top), nothing crashes or corrupts data, and the caption LLM still receives the factual headline (e.g. "New club record 55.10") alongside the wrong angle directive — so the miscue is the angle label in the review UI and the "Post angle: recap_mention" framing line in the caption prompt, not a total loss of the club-record story.

### F04 — Profile-priority multiplier plus min(1.0, ...) clamp saturates the top of the ranking — order among a club's boosted cards becomes input-order-arbitrary, even at the sanctioned x2.0 maximum, and the legacy profile path has no clamp at all

- **Where:** `legacy/swim_content_v5/ranker.py`:292
- **Category:** Correctness  ·  **Verdict:** CONFIRMED
- **What's wrong:** _compute_priority applies the club multiplier as `priority = min(1.0, max(0.0, base_priority * profile_multiplier))`. Any multiplier that pushes several cards past 1.0 collapses their scores to exactly 1.0, erasing the real strength difference between them; the subsequent single-key sort (line 404) then orders them purely by detector emission order. Precisely the achievement types a club flags as most important are the ones whose relative ranking becomes arbitrary. This is reachable through the clamped AI path: brand/derived.py:_norm_priorities allows up to 2.0 (including a '_default' key applying to every type), and at a national meet base priorities of 0.77/0.74/0.66 all exceed 0.5, so x2.0 flattens all of them. The legacy ClubProfile.get_achievement_priority path (web/club_profile.py:293) has NO clamp at all — any float including 100, 0, or negative passes straight through. A multiplier of 0 zeroes priority but the type-override band ignores it, leaving a suppressed gold still banded ELITE. [Reported independently by v5-ranker-logic and determinism-edge-probe audits.]
- **Failure scenario:** A club sets achievement_priorities['pb_*']=2.0 (or the AI derives a x2 default boost); at their next meet three PB cards all clamp to 1.0 and the #1 card in the pack — and the reel top-N selection — is whichever swim the parser happened to emit first, not the biggest PB.
- **Evidence:** Probe with profile multiplier 5.0 at national meet: pb_confirmed (weaker) and pb_magnitude_huge (stronger, 5% drop) both -> priority=1.0; ranking [weak, strong] returns weak first, ranking [strong, weak] returns strong first — order flips with input order. Second probe: no profile: [(1,'BARRIER',0.77),(2,'GOLD',0.74),(3,'HUGE',0.66)]; x2.0 fwd: all 1.0 in input order; x2.0 rev: reversed input reverses ranks. x100 legacy path: input order decides rank 1. brand/derived.py:315 clamps AI values to max(0.3, min(2.0, f)); club_profile.py _as_float has no clamp.
- **Refinement from verification:** Two refinements: (1) the unclamped legacy ClubProfile path is real code but unreachable through the product UI — no route writes profile.achievement_priorities; it requires hand-edited profile JSON (seed defaults max out at 1.5, which nonetheless already saturates: 1.3 x first_sub_barrier base 0.775 at a national meet clamps to 1.0 with the shipped defaults). (2) "suppressed gold still banded ELITE" is the deliberate type-floor OR branch in _assign_quality_band (line 298) — arguably intended design rather than a defect, though ranking position does drop.

### F05 — club_record is absent from every quality-band type override, so at club/county meets the club's biggest moment bands below any gold medal

- **Where:** `legacy/swim_content_v5/ranker.py`:296
- **Category:** Correctness  ·  **Verdict:** CONFIRMED
- **What's wrong:** The ELITE override list is ('medal_gold', 'first_sub_barrier', 'pb_magnitude_huge'); club_record — the type whose magnitude was hand-set to 1.1 as 'the highest-emotion club moment' — is in none of the band override lists. Its priority tops out ~0.77 (< 0.80 ELITE threshold) even at a national meet with full confidence, so it bands STRONG while every medal_gold bands ELITE. The two ranking surfaces then contradict each other: priority order puts club_record first, band/post-type order puts gold first, and builder routing (main_feed vs stories buckets) follows the band.
- **Failure scenario:** A swimmer breaks a 20-year club record at the club gala; the content pack files it under 'stories' while a routine gold from the same gala takes the main-feed slot.
- **Evidence:** Probe, national meet, conf 0.95: club_record -> rank=1, priority=0.77, band=strong; medal_gold -> rank=2, priority=0.745, band=elite. At a club-level meet club_record priority is ~0.65 -> STRONG -> PostType.STORY while gold is ELITE -> MAIN_FEED.
- **Refinement from verification:** One refinement: the finding says the band/bucket contradiction bites "at club/county meets", but the grouped-pack bucket routing (content_pack/builder.py:237-246) consults only quality_band, never meet level — so club_record lands in the stories bucket while gold takes main_feed at ALL meet levels, including national (where suggested_post_type=MAIN_FEED but the bucket is still "stories"). The club-meet case additionally degrades suggested_post_type itself to STORY.

### F09 — Page-level course detection flattens mixed LC/SC profile pages to one course, filing short-course baselines under long-course keys (and defaulting unknown pages to LC)

- **Where:** `src/mediahub/pb_discovery/parse_pbs.py`:95
- **Category:** Correctness  ·  **Verdict:** CONFIRMED
- **What's wrong:** _detect_course runs once on the whole page text and _heuristic_extract_pbs stamps that single course on every row. A page with both a 'Long Course' and a 'Short Course' table (the normal layout for swimmingresults/rankings profiles) returns 'LC' for everything because _LC_RE is checked first; with no marker at all the default is 'LC'. pb_bridge then keys the (typically faster) SC times as <dist><stroke>LC and leaves the SC key empty. The interpreter path inherits the same flaw via page_course fallback (parse_pbs.py lines 202, 212) when event.course is None.
- **Failure scenario:** Swimmer's true LC PB is 1:00.10, SC PB 57.80. They swim 59.60 LC — a genuine long-course PB — but the detector compares against the mis-filed 57.80 SC time and reports 'not faster than prior PB'; meanwhile SC swims have no baseline at all, so short-course PBs are never detected.
- **Evidence:** Probe PROBE 4: _detect_course over a page containing both 'Long Course' and 'Short Course' -> 'LC'; both extracted rows (LC 1:00.10 and SC 57.80) come back course='LC'; _detect_course with no marker -> 'LC'. Bridged, key 100FRLC holds [57.80, 60.10] so best LC baseline = the SC time.
- **Refinement from verification:** The claim that this hits "the normal layout for swimmingresults/rankings profiles" is wrong for production: swimmingresults.org personal-best pages are parsed by src/mediahub/swimmingresults/parse.py, which handles per-section course headings correctly and is the PRIMARY baseline source in pipeline_v4. parse_pbs.py only parses pages fetched by the SECONDARY pb_discovery gap-filler (swimmers not resolvable on swimmingresults.org, and only when an LLM provider is configured). Within that scope the flaw is fully real — and notably the interpreter path (not just the heuristic fallback) flattens mixed pages to LC, since the interpreter fails to attach course from section headings and falls back to page_course. Impact is false negatives (missed LC/SC PB detections, wrong-course audit baselines); safe_to_post gating prevents fabricated PB claims.

### F11 — All interactive surfaces key workflow state on bare achievement.swim_id, so duplicate swim_ids collide (approve/reject/caption on one card silently applies to its twin) while the deduped '~n' stub cards are unresolvable by every card-id lookup route
