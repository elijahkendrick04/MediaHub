## 1. Context and motivation

Phase 1.6 of the MediaHub roadmap (merged in PR #81) opens an
Adaptive Theming Engine that re-skins the entire website to a
club's brand colour the moment they accept the brand-DNA capture
on the `/organisation/setup` screen. The user-facing promise ‚Äî
"every single page, no matter how ugly the seed, looks
professional" ‚Äî rests on a precondition: the website's CSS must
reference a small set of *semantic role tokens* rather than
scattered hex values, because you cannot re-skin what is
hardcoded.

Today, MediaHub's main visual code lives in two CSS strings:

1. `BASE_CSS` ‚Äî a 1,740-line block at `src/mediahub/web/web.py:1472`
   defining the "Podium After Dark" theme (pit-wall black, lane-
   marker yellow `#D4FF3A`, medal gold `#F4D58D`, paper-cream ink
   `#F5F2E8`).
2. `RESPONSIVE_GUARDRAILS_CSS` ‚Äî a 440-line module at
   `src/mediahub/web/responsive_guardrails.py` appended to
   `BASE_CSS` for dvh/svh/lvh, container queries, safe-area
   insets, touch-target sizing, fluid type, motion preferences.

The existing token vocabulary is good but flat: ~30 tokens with
no hierarchy. The 141 unique hex literals and 203 rgba() literals
scattered through inline HTML `style="‚Ä¶"` strings ‚Äî emitted by
f-string templates ‚Äî bypass that vocabulary entirely. Stage A's
job is to (i) inventory every escape, (ii) introduce a Material-3-
style 3-tier vocabulary above the existing tokens, and (iii)
register every animatable colour token as a typed CSS custom
property so the Stage E cascade animation actually interpolates
instead of snapping.

The non-negotiable constraint is "no behaviour change" ‚Äî the site
must look pixel-identical after Stage A. Every existing token
name keeps working. The 678 passing tests keep passing. Stages
B‚ÄìJ will build on this foundation; A is foundation only.

## 2. Current state audit

The CSS-in-Python pattern means hardcoded colours can appear in
three places:

- **Inside `BASE_CSS`** ‚Äî most already abstracted into `--lane`,
  `--medal`, `--ink`, etc. Some literals remain inside box-shadow
  values, gradient stops, and the token-definition block itself
  (these are primitive seeds).
- **Inside f-string template literals** ‚Äî Python code like
  `f'<div style="background:rgba(212,255,58,0.15);‚Ä¶">'`. ~80 of
  these exist. Each one inlines colour values that should refer
  to a token. These are the highest-risk lookalike values.
- **Inside renderer modules** ‚Äî `graphic_renderer/`, `remotion/`
  ‚Äî out of scope for Stage A (Stages G handle the shared JSON).

The repository already has discipline: `--accent` is an alias of
`--lane`, `--panel` is an alias of `--surface`. The work in A2 is
to ratify that pattern with proper tier-1 primitives and tier-2
role tokens, then re-alias the existing names without changing
their resolved values.

The CI gates (`test_responsive_guardrails.py`,
`test_responsive_meta.py`, stylelint on
`graphic_renderer/layouts/*.css`) check structural invariants
that don't care about colour token names, so I can rename and
add freely as long as I preserve the existing names too.

## 3. A1 ‚Äî Inventory methodology

**Tool:** `scripts/inventory_colors.py` ‚Äî walks `web.py`
line-by-line, classifies every colour-literal match, emits a CSV
and summary report. Regex patterns:

- `#[0-9a-fA-F]{3,4}\b` and `#[0-9a-fA-F]{6,8}\b`
- `rgba?\(‚Ä¶\)`, `hsla?\(‚Ä¶\)`, `oklch\(‚Ä¶\)`, `oklab\(‚Ä¶\)`
- 16 CSS named colours used by Stylelint

Each match captures: file, line, full match, ¬±40 char context,
and a *classification*:

- `token_definition` ‚Äî inside a `:root` block. Expected and safe.
- `shadow_alpha` ‚Äî inside `box-shadow:` / `text-shadow:`.
- `gradient_stop` ‚Äî inside `linear-gradient(` etc.
- `inline_fstring` ‚Äî inside Python f-strings emitting HTML
  `style="‚Ä¶"`. **Migration targets.**
- `selection_color` ‚Äî `::selection { background: ‚Ä¶ }`.
- `js_string` ‚Äî inside JS template literals.
- `unclassified` ‚Äî manual triage.

Output: `docs/stage_a_color_inventory.csv` + stdout summary.
Expected: ~50 unique hex values, ~90 unique rgba, ~340
occurrences total, ~80 in `inline_fstring`.

## 4. A2 ‚Äî Three-tier DTCG token architecture

**Tier 1 ‚Äî Primitives (`--mh-prim-*`).** Raw values, no
semantics. 13-step tonal palettes per role + status anchors. For
Stage A, values are generated by hand from existing `--lane`,
`--medal`, `--bg`, `--ink` constants ‚Äî ratifying the existing
palette as a tonal ramp, not redesigning it. Stage B replaces
with HCT-derived values.

**Tier 2 ‚Äî Semantic role tokens (`--mh-*`).** The ~25 MD3-style
tokens. Reference primitives, never raw hex.

```

--mh-on-surface, --mh-on-surface-variant, --mh-on-surface-muted
--mh-primary, --mh-on-primary
--mh-primary-container, --mh-on-primary-container
--mh-secondary, --mh-on-secondary
--mh-tertiary, --mh-on-tertiary
--mh-tertiary-container, --mh-on-tertiary-container

--mh-error, --mh-on-error
--mh-success, --mh-warning, --mh-info
--mh-focus

```

**Tier 3 ‚Äî Component tokens.** Deferred per Nathan Curtis's
"promote across 3+ components" rule. Zero introduced in Stage A.

**Legacy aliases.** Every existing token re-points at a tier-2
role:

  --info-bg

Cascade resolves at use time ‚Üí every existing `var(--lane)`
callsite works unchanged. Zero edits to the 879 existing `var(--*)`
references.

**F-string hardcode migration.** The ~80 inline-fstring hex/rgba
literals from A1 ‚Üí replaced with `var(--mh-*)` references. Tinted
rgbas (e.g. `rgba(212,255,58,0.15)` = 15% lane yellow) ‚Üí either
the new tier-2 alias on the legitimate site, or kept as the
fallback while the relative-colour syntax is gated on `@supports`
(Stage E ships the modern path).

## 5. A3 ‚Äî `@property` registration

Without `@property`, CSS custom properties are untyped strings
and `transition: --bg 600ms` silently snaps. With `@property
syntax: "<color>"`, the browser knows the value is a colour and
can interpolate.

Stage A registers every tier-2 role token (~25) plus the legacy
aliases `--accent` and `--bg` (the most-referenced animation
handles in the Stage E plan). Tier-1 primitives are not
registered ‚Äî constant per theme, never animated.

```css
@property --mh-surface {
  syntax: "<color>";
  inherits: true;
  initial-value: #0A0B11;
}
```

`initial-value` must be a fully-resolved colour (spec disallows
`var()` in `initial-value`). Each set to the resolved value of
the legacy token it replaces. `inherits: true` correct for theme
tokens.

`@property` is Baseline Widely Available (94%+); older browsers
ignore the at-rule silently.

## 6. Cascade order and backward compatibility

Final cascade: `THEME_TOKENS_CSS` ‚Üí `BASE_CSS` (with reworked
`:root` referencing tier-2 tokens) ‚Üí `RESPONSIVE_GUARDRAILS_CSS`.

Backward compatibility is absolute. Test asserts old and new
`:root` blocks resolve to identical per-token values.

## 7. Code organisation

New file: `src/mediahub/web/theme_tokens.py`. Mirrors
`responsive_guardrails.py` pattern ‚Äî exports a single string
`THEME_TOKENS_CSS`. Not a Python data structure; Stage B
introduces the data layer.

In `web.py`, `BASE_CSS` stays put. The `:root` block inside it is
rewritten so previously-raw hex becomes `var(--mh-prim-*)`. At
the import site (~line 3220), a single concat prepends
`THEME_TOKENS_CSS`:

```python
from mediahub.web.theme_tokens import THEME_TOKENS_CSS as _MH_TT_CSS
from mediahub.web.responsive_guardrails import RESPONSIVE_GUARDRAILS_CSS as _MH_RG_CSS
BASE_CSS = _MH_TT_CSS + BASE_CSS + _MH_RG_CSS
```

F-string migration via grouped `Edit` calls per literal.
