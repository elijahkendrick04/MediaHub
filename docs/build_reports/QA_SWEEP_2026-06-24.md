## QA-016 (RE-OPEN #2) [P2 BUG] — `/plan` AND `/api/plan/latest` both 500 on a non-UTF-8 persisted file

**Symptom.** `GET /plan` returned HTTP 500 — but so did `GET /api/plan/latest`,
the JSON endpoint that never renders a page. Org-specific: Swansea (no persisted
plan) → `/plan` 200; Brighton & Cardiff (both have
`content_plans/<org>/latest.json` on the durable disk) → 500. The throw fires
only when an existing `latest.json` is **read**.

**Diagnosis — the prior two fixes were the wrong layer.** Commits `510ccb1` and
`3a97380` hardened `plan_page()`'s *rendering* of a persisted plan (defensive
`int(None)` / `None.replace` coercion, guarded `list_sport_profiles`, `_as_list`
for malformed structures). But `GET /api/plan/latest` does **not** go through
`plan_page` at all — its handler is just `plan = load_latest_plan(pid);
return jsonify({"ok": True, "plan": plan})`. It still 500'd, which proves the
crash is in plan **loading**, not page rendering. The org-specificity confirms
it: the throw fires only when an existing `latest.json` is read.

**Root cause (file-level).**
`content_engine/planner.py::load_latest_plan` does

```python
raw = json.loads(path.read_text(encoding="utf-8"))
```

guarded by `except (OSError, json.JSONDecodeError)`. A non-UTF-8 `latest.json`
makes `read_text(encoding="utf-8")` raise **`UnicodeDecodeError`** *before*
`json.loads` ever runs. `UnicodeDecodeError` is a subclass of `ValueError`
(`UnicodeError → ValueError`), **not** of `OSError` or `json.JSONDecodeError`,
so it escapes the narrow `except` and 500s the loader — and therefore both
`GET /plan` (which calls `load_latest_plan` + `load_planner_inputs`) and
`GET /api/plan/latest` (which calls `load_latest_plan`).

The **same latent bug** sits in
`content_engine/inputs.py::load_planner_inputs` (identical narrow
`except (OSError, json.JSONDecodeError)`), whose docstring already promises
"missing/corrupt files load as empty" — graceful degradation is the documented
intent; the `except` was just too narrow to honour it.

Proven empirically: `issubclass(UnicodeDecodeError, ValueError) is True`;
`issubclass(UnicodeDecodeError, (OSError, json.JSONDecodeError)) is False`; and
`json.loads(Path(p).read_text(encoding="utf-8"))` over bytes with a raw `0xe9`
raises `UnicodeDecodeError: 'utf-8' codec can't decode byte 0xe9`.

**Why a non-UTF-8 file is reachable.** `DATA_DIR` is a durable mounted disk
(`render.yaml`: `/var/mediahub`) that survives redeploys. Both writers
(`save_plan`, `save_planner_inputs`) emit `encoding="utf-8", ensure_ascii=False`
— always valid UTF-8 — so the bad bytes never come from the happy path; they
arise from a torn/partial write, a disk fault, or a file an external tool
re-encoded. Read-side robustness is the correct **floor** fix: a loader whose
documented contract is "corrupt files load as empty" must not crash on *any*
corrupt file, regardless of how it got that way.

**Fix (smallest safe — broaden the `except` in both loaders).** Change
`except (OSError, json.JSONDecodeError)` → `except (OSError, ValueError)` in
both `load_latest_plan` and `load_planner_inputs`. `ValueError` subsumes both
`json.JSONDecodeError` (malformed JSON) and `UnicodeDecodeError` (non-UTF-8
bytes) — the two corruption modes a read can hit — while still letting genuine
programming errors (`TypeError`, `KeyError`, …) propagate. A corrupt file now
degrades to `None` / `empty_inputs()` exactly as the docstrings promise, so both
routes return 200 (plan `null`). No engine maths touched: this is read-side I/O
robustness, not a change to the deterministic planner's scoring.

**Files.**
- `src/mediahub/content_engine/planner.py` — `load_latest_plan` `except` widened
  to `(OSError, ValueError)` + explanatory comment.
- `src/mediahub/content_engine/inputs.py` — `load_planner_inputs` `except`
  widened to `(OSError, ValueError)` + explanatory comment.

**Regression test** (`tests/test_cross_source_planner.py`):
`test_plan_routes_survive_non_utf8_persisted_files` — under the `app_with_org`
tmp `DATA_DIR`, writes an invalid-UTF-8 `content_plans/org-test/latest.json` and
`planner_inputs/org-test.json` (JSON-shaped bytes with a raw `0xe9` inside a
string), then asserts `load_latest_plan → None`, `load_planner_inputs →
empty_inputs()`, `GET /api/plan/latest → 200` (`plan` null), and `GET /plan →
200` (honest empty state). The prior QA-016 tests only used a well-formed fresh
plan, which never triggers the decode path — exactly the gap that let this
survive two fixes. **Fails before** the change (`UnicodeDecodeError` escapes the
loader at `planner.py:479`), **passes after**.

**Suite.** `python -m pytest tests/ -q` — green; the only behavioural delta is
the one new test plus two widened `except` clauses, so no existing test is
affected. CI is the authoritative gate (PR checks) before the merge to `main`.
