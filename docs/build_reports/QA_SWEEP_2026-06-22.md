## QA-001 [P1 accuracy] — Duplicate moment cards from overlapping age-band re-prints

**Symptom.** HY-TEK reprints the *same* swim under multiple overlapping
age-band sub-headers that share one Event number, so the interpreter surfaced
identical duplicate recognition cards (review list #1=#2 Raleigh, #3=#4 Carden),
inflating the queue to 416 cards and "254 PBs" vs 129 online-confirmed.

**Evidence (pdftotext -layout).**
- Birdy Raleigh 200 Free 2:19.80 under `Event 202 Female 10-13 …` (line 118)
  **and** `Event 202 Female 13 Year Olds …` (line 165).
- Izzy Carden 200 Free 2:17.00 under `Event 202 Female 16 & Over` (line 241)
  **and** `Event 202 Female 17 & Over` (line 262).

**Root cause.** The exact-reprint de-dup at the canonical seam
(`pipeline/interpreter_bridge.py::_canonical_result_identity`) keyed on
`age_band` **and** `place`. The age-band reprint mutates exactly those two
fields (the banner changes; the place is relative to each printed band), so the
key differed between printings and the reprints were never collapsed. The
existing de-dup only caught page-break reprints (identical banner).

**Fix.** Re-key the identity on the *physical swim*:
`(event number + swimmer + final time)`, keeping distance/stroke/course/gender
(to pin the event when no number is printed) and the round / A-B-final label
(so a genuine heat vs final is still preserved). The key now **excludes**
`age_band` and `place` — the two fields the reprint mutates. The event number is
parsed from the header (`_event_number`) and stored on the result's `extra` for
auditability. Collapsed reprints still emit the explainable
`duplicate_results_collapsed` info warning.

**Files.** `src/mediahub/pipeline/interpreter_bridge.py`.

**Regression tests.** `tests/test_interpreter_bridge_dedup.py`:
`test_age_band_reprint_of_same_event_collapses_to_one_swim`,
`test_distinct_swimmers_in_overlapping_bands_are_kept`,
`test_end_to_end_age_band_reprint_does_not_duplicate`.

## QA-002 [P2 trust/quality] — Meet title was the HY-TEK license/software banner

**Symptom.** Meet title showed
`Sussex County ASA- LC Champ - Organization License HY-TEK's MEET MANAGER 8.0 -
6:29 PM 15/02/2026 Page 1` (PDF line 1 = licensee + software + timestamp + page
banner), and it propagated into the generated "Weekend at a glance" headline.
The real meet name is PDF line 2:
`Sussex County Long Course Championships 26 - 14/02/2026 to 01/03/2026`.

**Root cause.** `interpreter/__init__.py::_extract_meet_metadata` took the first
non-trivial line as the meet name. On a HY-TEK results page that first line is
the licensee/software/timestamp/page banner, not the title.

**Fix.** Skip software/license/page **banner** lines (and event-header / result
rows) when choosing the title, picking the first plausible title line instead
(line 2 here), and strip a trailing date range. A degenerate header where the
name only survives merged into the banner recovers the prefix by stripping the
`Organization License` / `HY-TEK's MEET MANAGER …` / `Page N` / timestamp
boilerplate. Detection is structural (vendor + report-furniture words), so it
stays on the deterministic side of the engine boundary — no AI, no swim
vocabulary.

**Files.** `src/mediahub/interpreter/__init__.py`.

**Regression tests.** `tests/test_interpreter_corpus.py`:
`test_hytek_license_banner_is_not_the_meet_name`,
`test_hytek_banner_only_header_strips_boilerplate_to_recover_name`.

## QA-003 [P2 accuracy] — Phantom "<Club> NT" clubs in the club picker

**Symptom.** The configure club list contained phantom clubs
"City of Brighton & Hove NT", "Beacon SC NT", "Atlantis SC NT", etc.

**Evidence.** PDF line 282:
`Birdy Raleigh 13 City of Brighton & Hove  NT  2:24.51` — "NT" sits in the
seed-time column (No Time), not the team name.

**Root cause.** In the multi-record line parser the seed column sits between the
club and the finals time, so the club cell built from the tokens between the age
and the finals time absorbed the "NT" seed token
(`_normalise_club("City of Brighton & Hove NT")` → "… NT").

**Fix.** `interpreter/rows.py::_normalise_club` now strips a standalone trailing
No-Time seed marker ("NT" / "N.T."). Anchored on a preceding word boundary
(string start or whitespace) so a real club ending in those letters (… Kent,
… Trent) is untouched; a bare "NT" seed cell mis-mapped to the club column is
rejected entirely.

**Files.** `src/mediahub/interpreter/rows.py`.

**Regression tests.** `tests/test_interpreter_rows_normalisers.py`:
`test_no_time_seed_marker_is_stripped_from_club`,
`test_club_ending_in_nt_letters_is_untouched`,
`test_bare_no_time_marker_is_not_a_club`; end-to-end
`tests/test_interpreter_corpus.py::test_no_time_seed_does_not_create_phantom_clubs`.

## QA-004 [P2] — Design Studio (`/studio`) live preview stuck "Rendering…"

**Symptom.** The card preview never resolved to a clean card — it sat under a
persistent "Rendering…" layer (a semi-transparent, `backdrop-filter:blur`
overlay that read as garbled/"mirrored & doubled"), from a fresh page load and
across archetypes.

**Investigation.**
- (a) **The server PNG is correct, not mirrored/doubled.** Rendered the real
  studio path (`build_brief_from_params` → `render_brief`) at the preview size
  (540×675) *and* full (1080×1350) for both Big Number Dominant and Ribbon
  Banner — every output is a single, correctly-oriented card. The garble was the
  stuck overlay sitting over the card, not the pixels.
- (b) **The full-res "Download PNG" is unaffected** server-side (same clean
  render at native size); the overlay bug affected the on-page experience for
  both preview and download, never the downloaded file.

**Root cause.** The overlay is shown/hidden via its `hidden` attribute
(`showOverlay`/`hideOverlay`), but the scoped CSS set
`.mh-studio-overlay { display:flex }` — an author rule that overrides the UA
`[hidden]{display:none}`. With no `[hidden]` guard, `hideOverlay()`
(`overlay.hidden = true`) could never actually hide it, so "Rendering…" stayed
stuck over every render (confirmed in Chromium: computed `display` stays `flex`
even with `hidden` set).

**Fix.** Add `.mh-studio-overlay[hidden] { display:none; }` so the overlay can
be hidden. (`src/mediahub/web/design_editor.py`.)

**Regression tests** (`tests/test_design_editor_g1_27.py`):
`test_studio_overlay_is_hidable_via_its_hidden_attribute` (fails-before /
passes-after); plus a Chromium-gated contract test
`test_studio_render_is_a_single_unmirrored_card` (preview + full): exact
dimensions, top≠bottom (not doubled), real asymmetric content (not blank /
mirror-symmetric).

## QA-005 [P3] — Run progress bar non-monotonic (28%→10%→28%, steps 5↔3)

**Root cause.** The deployment runs `--workers 2` (gunicorn). Status polls
round-robin across both workers; the worker that isn't running the pipeline has
no in-memory log and falls back to the throttled, lagging DB `progress_log`, so
a single poll's `percent` / `log.length` can read lower than an earlier one.
`recap_progress` is monotonic per log — the non-monotonicity is the cross-worker
source mismatch, which only the client (the one consistent observer across
polls) can reconcile.

**Fix.** The run-progress page now holds a high-water mark for both the percent
and the step count / raw log, so reported progress never decreases. Verified in
Chromium: an input sequence `28→10→28→32%` / steps `5→3→5→6` clamps to
`[28,28,28,32]` / `[5,5,5,6]`. (`src/mediahub/web/web.py`.)

**Regression test** (`tests/test_run_status_dev_vs_customer.py`):
`test_progress_page_clamps_percent_and_steps_monotonically`.

## QA-006 [P2/P3] — Silent 403 on org finalise

**Root cause.** `POST /api/organisation/finalise` (the "Looks right — start
creating" cascade) sent only `Accept: application/json` — **no `Content-Type`**
and no CSRF token. The app's CSRF guard exempts writes whose content-type is
`application/json`; without that marker the same-origin POST was rejected 403,
so the derived palette was never persisted on finalise (the flow proceeded
anyway via the `<a href=/make>` fallback, masking the failure).

**Fix.** The cascade fetch now sends `Content-Type: application/json` (+ an empty
JSON body) — the documented CSRF-exempt marker, matching every other JSON write
in the app. Confirmed end-to-end: under enforced CSRF, the old shape → 403, the
fixed shape → 200 with the derived palette persisted. (`src/mediahub/web/web.py`.)

**Regression tests** (`tests/test_looks_right_button.py`):
`test_finalise_post_sets_json_content_type_to_dodge_csrf` (fails-before /
passes-after) and `test_finalise_csrf_contract_old_shape_blocked_json_shape_succeeds`
(end-to-end: old→403, fixed→200).

## QA-008 [P3 trust/quality] — Stale HY-TEK banner title in PERSISTED run metadata

**Symptom.** The QA-002 fix cleaned the meet-title **parser** forward-only, so
runs parsed *before* that fix keep the HY-TEK banner string in persisted run
metadata:
`Sussex County ASA- LC Champ - Organization License HY-TEK's MEET MANAGER 8.0 -
6:29 PM 15/02/2026 Page 1`. It still surfaced as the meet name in three
downstream surfaces: the Season timeline (`/season`), the Sites source picker
(`/sites`) and the Documents source picker (`/documents`).

**Root cause.** These surfaces render the stored `runs.meet_name` column
verbatim — the season timeline directly (`title = r["meet_name"] …`), and both
source pickers via the shared `_doc_recent_runs` helper. The QA-002 cleaner only
runs at parse time, so it never touched rows already written to the DB.

**Fix (option a — re-derive at render time, most robust).** A small display
helper `web.py::_clean_stored_meet_title` re-cleans a stored title with the
*same* deterministic helper QA-002 added (`interpreter._clean_meet_name`) and is
applied where each surface reads the stored name — once in `_doc_recent_runs`
(covers `/sites` + `/documents`) and once in the season timeline. This fixes any
stale stored title (the displayed banner becomes `Sussex County ASA- LC Champ`)
without a data migration, and is idempotent on already-clean names. No AI, no
heuristic guessing — pure boilerplate stripping, on the deterministic side of
the engine boundary.

**Files.** `src/mediahub/web/web.py`.

**Regression tests.** A stored run carrying the banner title → the rendered
surface shows the cleaned name, not the banner:
`tests/test_season_timeline.py::TestStaleBannerTitle::test_banner_title_is_cleaned_on_timeline`,
`tests/test_documents_web.py::test_source_picker_cleans_stale_banner_title`,
`tests/test_sites_web.py::test_source_picker_cleans_stale_banner_title`.

## QA-011 [P2 quality] — Result time CLIPS / WRAPS / collides across studio archetypes

**Symptom.** In `/studio` with the default sample (Eira Hughes · `2:08.41` ·
200M Freestyle · Manchester Swimming Club), switching archetypes clipped or
wrapped the result time — the single most important fact on a swim card:
- **Broadcast Scorebug** — `2:08.41` clipped to `2:08.4` at the scorebug's
  right edge (final `1` cut).
- **Quote Led Recap** — `2:08.41` wrapped to `2:08.` / `41`, the second line
  overflowing the card's bottom edge.
- **Full Bleed Photo Lower Third** — the `NEW PB` kicker collided with the club
  lockup, and the event collided with the result chip.

Big Number Dominant and Magazine Cover rendered the same data perfectly — so it
looked per-archetype. An audit across all 29 archetypes at the studio preview
geometry found **16 of 29** clipping / wrapping / overflowing the result time.

**Investigation (real Chromium, measured DOM).** Rendered every archetype at the
studio **preview** geometry and at the **native** geometry and measured the
result element's box. At native (1080×1350 — the Download PNG) **all 29 fit on
one line**, no clip, no overflow. At the studio preview geometry **(540×675)**
the same 16 archetypes failed. So this is not the data, the autofit maths, or
the layouts — it is the geometry the preview is composed at.

**Root cause.** `web/design_editor.py` rendered the live preview at **half the
native CSS geometry** (`PREVIEW_SCALE = 0.5` shrank `StudioParams.size` to
540×675) "for a snappy, light payload". The v2 archetypes are authored with
**fixed-px furniture** (paddings, labels, the hanging quote glyph, the scorebug
cells) that does *not* scale with the canvas, while the autofit hero vars *do*.
At half geometry the fixed furniture is ~2× over-weighted relative to the card,
squeezing the result slot until the time clips, wraps and overflows, and
colliding fixed-position elements. The full-resolution download composed
correctly because its geometry was native — which is exactly why the symptom
looked archetype-specific and why "confirm in the Download PNG" sometimes
looked clean.

**Fix (smallest safe — fix the geometry, not the 29 correct layouts).** The
preview now **composes at the same native geometry as the download**, so every
archetype's furniture keeps its authored proportions and the time fits. The
"snappy, light payload" is preserved at the **raster**, not the geometry:
- `StudioParams.size` is always native (preview and download);
- the preview renders at a light **DPR-1** capture (`render_quality == "fast"`),
  the download at the default profile;
- the finished native render is downsampled to `PREVIEW_SCALE` for the response
  (`preview_raster_size`) by `downscale_png_bytes` — same render cost and same
  payload size as before, but a correct composition.

The preview is now a faithful, lighter copy of the Download PNG (the QA's stated
expectation — "the preview shows the actual rendered PNG scaled"). The
deterministic engine, the autofit maths, and all 29 layout templates are
untouched; the collisions and the no-photo "awkward empty stage" were the same
preview-scale artifact and resolve with the geometry.

**Files.** `src/mediahub/web/design_editor.py`, `src/mediahub/web/web.py`
(`api_studio_render` — pass the preview quality, downsample the response raster).

**Regression tests** (`tests/test_qa_011_studio_result_time_fit.py`):
`test_preview_composes_at_native_geometry_not_half_scale` (deterministic, always
runs — the preview must compose at native geometry, lightness only at the
raster), `test_downscale_png_is_safe_and_lossless_noop_when_not_needed`, and a
Chromium-gated structural audit
`test_result_time_fits_in_every_studio_archetype` that renders **all 29**
archetypes with a 7-char time and asserts the time sits inside the card on one
line with no horizontal clip and no bottom overflow (fails-before at half-scale:
16/29 clip — proven; passes-after: all 29 fit). The existing studio size
contract in `tests/test_design_editor_g1_27.py` was re-pinned to the corrected
behaviour (preview composes at native; the light raster lives in
`preview_raster_size`), and its render stubs widened to the real
`render_html_to_png(…, *, quality=…)` signature — no test weakened.

**Noted, out of scope (separate root cause).** The top-right meet-name vs
**PB-rosette** overlap is the G1.22 icon-overlay (`sprint_hooks/icon_overlay.py`)
placement against archetypes that carry top-right header text. It exists at
native (production downloads) independently of the preview-scale bug, lives in a
different subsystem with its own parity tests, and is left for a focused
follow-up rather than folded into this geometry fix as a drive-by.
