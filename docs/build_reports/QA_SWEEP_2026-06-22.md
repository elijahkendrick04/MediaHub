## QA-001 [P1 accuracy] Рђћ Duplicate moment cards from overlapping age-band re-prints

**Symptom.** HY-TEK reprints the *same* swim under multiple overlapping
age-band sub-headers that share one Event number, so the interpreter surfaced
identical duplicate recognition cards (review list #1=#2 Raleigh, #3=#4 Carden),
inflating the queue to 416 cards and "254 PBs" vs 129 online-confirmed.

**Evidence (pdftotext -layout).**
- Birdy Raleigh 200 Free 2:19.80 under `Event 202 Female 10-13 Рђд` (line 118)
  **and** `Event 202 Female 13 Year Olds Рђд` (line 165).
- Izzy Carden 200 Free 2:17.00 under `Event 202 Female 16 & Over` (line 241)
  **and** `Event 202 Female 17 & Over` (line 262).

**Root cause.** The exact-reprint de-dup at the canonical seam
(`pipeline/interpreter_bridge.py::_canonical_result_identity`) keyed on
`age_band` **and** `place`. The age-band reprint mutates exactly those two
fields (the banner changes; the place is relative to each printed band), so the
key differed between printings and the reprints were never collapsed. The
existing de-dup only caught page-break reprints (identical banner).

**Fix.** Re-key the identity on the *physical swim*:
`(event number + swimmer + final time)`, keeping distance/stroke/course/gender
(to pin the event when no number is printed) and the round / A-B-final label
(so a genuine heat vs final is still preserved). The key now **excludes**
`age_band` and `place` Рђћ the two fields the reprint mutates. The event number is
parsed from the header (`_event_number`) and stored on the result's `extra` for
auditability. Collapsed reprints still emit the explainable
`duplicate_results_collapsed` info warning.

**Files.** `src/mediahub/pipeline/interpreter_bridge.py`.

**Regression tests.** `tests/test_interpreter_bridge_dedup.py`:
`test_age_band_reprint_of_same_event_collapses_to_one_swim`,
`test_distinct_swimmers_in_overlapping_bands_are_kept`,
`test_end_to_end_age_band_reprint_does_not_duplicate`.

## QA-002 [P2 trust/quality] Рђћ Meet title was the HY-TEK license/software banner

**Symptom.** Meet title showed
`Sussex County ASA- LC Champ - Organization License HY-TEK's MEET MANAGER 8.0 -
6:29 PM 15/02/2026 Page 1` (PDF line 1 = licensee + software + timestamp + page
banner), and it propagated into the generated "Weekend at a glance" headline.
The real meet name is PDF line 2:
`Sussex County Long Course Championships 26 - 14/02/2026 to 01/03/2026`.

**Root cause.** `interpreter/__init__.py::_extract_meet_metadata` took the first
non-trivial line as the meet name. On a HY-TEK results page that first line is
the licensee/software/timestamp/page banner, not the title.

**Fix.** Skip software/license/page **banner** lines (and event-header / result
rows) when choosing the title, picking the first plausible title line instead
(line 2 here), and strip a trailing date range. A degenerate header where the
name only survives merged into the banner recovers the prefix by stripping the
`Organization License` / `HY-TEK's MEET MANAGER Рђд` / `Page N` / timestamp
boilerplate. Detection is structural (vendor + report-furniture words), so it
stays on the deterministic side of the engine boundary Рђћ no AI, no swim
vocabulary.

**Files.** `src/mediahub/interpreter/__init__.py`.

**Regression tests.** `tests/test_interpreter_corpus.py`:
`test_hytek_license_banner_is_not_the_meet_name`,
`test_hytek_banner_only_header_strips_boilerplate_to_recover_name`.

## QA-003 [P2 accuracy] Рђћ Phantom "<Club> NT" clubs in the club picker

**Symptom.** The configure club list contained phantom clubs
"City of Brighton & Hove NT", "Beacon SC NT", "Atlantis SC NT", etc.

**Evidence.** PDF line 282:
`Birdy Raleigh 13 City of Brighton & Hove  NT  2:24.51` Рђћ "NT" sits in the
seed-time column (No Time), not the team name.

**Root cause.** In the multi-record line parser the seed column sits between the
club and the finals time, so the club cell built from the tokens between the age
and the finals time absorbed the "NT" seed token
(`_normalise_club("City of Brighton & Hove NT")` Рєњ "Рђд NT").

**Fix.** `interpreter/rows.py::_normalise_club` now strips a standalone trailing
No-Time seed marker ("NT" / "N.T."). Anchored on a preceding word boundary
(string start or whitespace) so a real club ending in those letters (Рђд Kent,
Рђд Trent) is untouched; a bare "NT" seed cell mis-mapped to the club column is
rejected entirely.

**Files.** `src/mediahub/interpreter/rows.py`.

**Regression tests.** `tests/test_interpreter_rows_normalisers.py`:
`test_no_time_seed_marker_is_stripped_from_club`,
`test_club_ending_in_nt_letters_is_untouched`,
`test_bare_no_time_marker_is_not_a_club`; end-to-end
`tests/test_interpreter_corpus.py::test_no_time_seed_does_not_create_phantom_clubs`.

## QA-004 [P2] Рђћ Design Studio (`/studio`) live preview stuck "RenderingРђд"

**Symptom.** The card preview never resolved to a clean card Рђћ it sat under a
persistent "RenderingРђд" layer (a semi-transparent, `backdrop-filter:blur`
overlay that read as garbled/"mirrored & doubled"), from a fresh page load and
across archetypes.

**Investigation.**
- (a) **The server PNG is correct, not mirrored/doubled.** Rendered the real
  studio path (`build_brief_from_params` Рєњ `render_brief`) at the preview size
  (540├Ќ675) *and* full (1080├Ќ1350) for both Big Number Dominant and Ribbon
  Banner Рђћ every output is a single, correctly-oriented card. The garble was the
  stuck overlay sitting over the card, not the pixels.
- (b) **The full-res "Download PNG" is unaffected** server-side (same clean
  render at native size); the overlay bug affected the on-page experience for
  both preview and download, never the downloaded file.

**Root cause.** The overlay is shown/hidden via its `hidden` attribute
(`showOverlay`/`hideOverlay`), but the scoped CSS set
`.mh-studio-overlay { display:flex }` Рђћ an author rule that overrides the UA
`[hidden]{display:none}`. With no `[hidden]` guard, `hideOverlay()`
(`overlay.hidden = true`) could never actually hide it, so "RenderingРђд" stayed
stuck over every render (confirmed in Chromium: computed `display` stays `flex`
even with `hidden` set).

**Fix.** Add `.mh-studio-overlay[hidden] { display:none; }` so the overlay can
be hidden. (`src/mediahub/web/design_editor.py`.)

**Regression tests** (`tests/test_design_editor_g1_27.py`):
`test_studio_overlay_is_hidable_via_its_hidden_attribute` (fails-before /
passes-after); plus a Chromium-gated contract test
`test_studio_render_is_a_single_unmirrored_card` (preview + full): exact
dimensions, topРЅаbottom (not doubled), real asymmetric content (not blank /
mirror-symmetric).

## QA-005 [P3] Рђћ Run progress bar non-monotonic (28%Рєњ10%Рєњ28%, steps 5Рєћ3)

**Root cause.** The deployment runs `--workers 2` (gunicorn). Status polls
round-robin across both workers; the worker that isn't running the pipeline has
no in-memory log and falls back to the throttled, lagging DB `progress_log`, so
a single poll's `percent` / `log.length` can read lower than an earlier one.
`recap_progress` is monotonic per log Рђћ the non-monotonicity is the cross-worker
source mismatch, which only the client (the one consistent observer across
polls) can reconcile.

**Fix.** The run-progress page now holds a high-water mark for both the percent
and the step count / raw log, so reported progress never decreases. Verified in
Chromium: an input sequence `28Рєњ10Рєњ28Рєњ32%` / steps `5Рєњ3Рєњ5Рєњ6` clamps to
`[28,28,28,32]` / `[5,5,5,6]`. (`src/mediahub/web/web.py`.)

**Regression test** (`tests/test_run_status_dev_vs_customer.py`):
`test_progress_page_clamps_percent_and_steps_monotonically`.

## QA-006 [P2/P3] Рђћ Silent 403 on org finalise

**Root cause.** `POST /api/organisation/finalise` (the "Looks right Рђћ start
creating" cascade) sent only `Accept: application/json` Рђћ **no `Content-Type`**
and no CSRF token. The app's CSRF guard exempts writes whose content-type is
`application/json`; without that marker the same-origin POST was rejected 403,
so the derived palette was never persisted on finalise (the flow proceeded
anyway via the `<a href=/make>` fallback, masking the failure).

**Fix.** The cascade fetch now sends `Content-Type: application/json` (+ an empty
JSON body) Рђћ the documented CSRF-exempt marker, matching every other JSON write
in the app. Confirmed end-to-end: under enforced CSRF, the old shape Рєњ 403, the
fixed shape Рєњ 200 with the derived palette persisted. (`src/mediahub/web/web.py`.)

**Regression tests** (`tests/test_looks_right_button.py`):
`test_finalise_post_sets_json_content_type_to_dodge_csrf` (fails-before /
passes-after) and `test_finalise_csrf_contract_old_shape_blocked_json_shape_succeeds`
(end-to-end: oldРєњ403, fixedРєњ200).

## QA-009 [P2/P3] Рђћ Media Library shows "store wasn't readable" instead of a clean empty state

**Symptom.** An organisation that has never uploaded media (repro: "Swansea
University Swimming Team", `000 ASSETS`) opens `/media-library` and the
asset-list section renders the alarming
*"Couldn't load library assets Рђћ the store wasn't readable. Uploads above still
work; if this persists, ask your operator to check the data volume."* It
**persists across deploys** (re-checked on a healthy build Рђћ not transient).
**Expected:** the clean empty state ("No assets uploaded yet. Drop a photo above
to get started.") when the store is simply empty / not-yet-created for the org.

**Root cause.** The page (`media_library_page`, `web/web.py`) wrapped the media
read in a blanket `except Exception Рєњ store_failed = True`, so **every** read
failure Рђћ including the benign "no store on disk yet" case Рђћ rendered the
corrupt-store recovery message. The asset DB (`media_library/store.py`) is
created lazily on first **write**; its default path
(`src/mediahub/data.db`) lives in the *application* directory, **not** under
`DATA_DIR`. On a deployment where that directory is read-only (Render's code
layer), the first `get_store()` / `store.list()` for a fresh org can't create
the file and raises `OSError` / `sqlite3.OperationalError("unable to open
database file")` rather than returning `[]`. The route mislabelled that as
corruption. (A genuinely empty *writable* store already returns `[]` and showed
the empty state correctly Рђћ only the raising path was mishandled.)

**Fix.** Distinguish *"missing but fine"* from *"genuinely corrupt"* by the one
honest signal available: **is the DB file actually on disk?** On a read failure,
if the store's `db_path` (or the module default, when the store couldn't even be
constructed) does **not** exist Рєњ there is nothing wrong, just nothing uploaded
Рєњ render the clean empty state. Only when the DB file **is** present but won't
read Рєњ keep `store_failed = True` and surface the "check the data volume"
recovery message. No new dependency, no engine touched, no AI/heuristic
fallback Рђћ a benign-empty store is reported honestly as empty, a corrupt one is
still reported honestly as corrupt. (`src/mediahub/web/web.py`.)

> Note (out of scope, recorded): the deeper smell is that the media store's
> default DB path is hardcoded to `src/mediahub/data.db` instead of being
> derived from `DATA_DIR`. Re-homing it under `DATA_DIR` would let uploads
> persist on read-only-code-dir deploys, but it's a data-location change across
> many call sites (uploads, deletes, DSR, video projects, org lifecycle) with
> migration implications Рђћ a separate, larger change, not folded into this
> empty-state display fix.

**Files.** `src/mediahub/web/web.py`.

**Regression tests** (`tests/test_media_library_empty_state.py`):
`test_missing_store_renders_empty_state_not_error` and
`test_store_construction_failure_with_no_db_renders_empty_state` (both
fail-before / pass-after Рђћ the empty state is shown, the error is not), plus the
guard `test_genuinely_corrupt_store_still_shows_recovery_message` (DB file
present but invalid still surfaces the recovery message Рђћ passes both ways).

# QA Sweep Рђћ 2026-06-22 (batch 5: QA-010)

A fifth live-site hand-off Рђћ a P1 in the Video Studio footageРєњreel flow. Same

## QA-010 [P1] Рђћ Video Studio footage upload returns HTTP 500 (footageРєњreel blocked at step 1)

**Symptom.** Opening `/video` and uploading a clean, valid 5-second 720├Ќ1280
portrait H.264 (yuv420p) + AAC MP4 (271 KB, well under the 50 MB-raised cap)
returned `POST /api/video/footage` **500**; the studio showed
`Upload failed: internal_error` and `No footage yet`. A volunteer could not make
a reel at all. Reproduced twice.

**Root cause.** The media-library store (`media_library/store.py`) Рђћ which the
footage-ingest path writes through (`video/ingest.py::ingest_footage_stream` Рєњ
`store.store_blob_stream` / `store.save`) Рђћ hardcoded its blob + `data.db`
defaults to the **package dir** (`Path(__file__).resolve().parents[1]`, i.e.
`/app/src/mediahub/uploads_v4/Рђд` and `/app/src/mediahub/data.db`) instead of
deriving them from `DATA_DIR`. On the deploy the code tree is **root-owned** and
the runtime user is **uid 10001** (`Dockerfile` chowns only `/app/runs_v4`,
`/app/uploads_v4`, `/app/.cache`, `/app/data` to `mediahub`; everything else
under `/app` stays root). Writing a footage blob there raised **`PermissionError`**
(a subclass of `OSError`), which the route did not catch Рђћ only `ValueError` was
handled Рђћ so it fell through to the global 500 handler that emits
`{"error": "internal_error"}`. The sibling Video-Studio store
(`video/projects.py::_data_dir`) and every other store (workflow schedule,
autonomy, review-comments, audio upload) already key off `DATA_DIR`; the media
library was the lone outlier, also violating the CLAUDE.md **DATA_DIR rule**
("all storage paths derived from `DATA_DIR`; never hardcode"). Any data that
*did* land in the package dir was also lost on redeploy.

**Fix.**
1. **Root cause** Рђћ `media_library/store.py` now resolves its default DB +
   uploads dir from `DATA_DIR` at construction time (`_data_dir()` /
   `_default_db_path()` / `_default_uploads_dir()`), mirroring `video/projects.py`
   exactly, with the same package-dir fallback only when `DATA_DIR` is unset.
   Footage/photo blobs now land on the writable persistent disk
   (`/var/mediahub/uploads_v4/media_library`, `/var/mediahub/data.db`). Explicit
   `db_path`/`uploads_dir` args (tests, callers) are unchanged.
2. **Honest error (no more `internal_error`)** Рђћ `api_video_footage_upload` now
   catches `OSError` Рєњ `{"error": "storage_failed", "message": Рђд}` (an honest,
   specific "couldn't be saved Рђћ storage full or unavailable" in the style of the
   meet-file/audio upload errors) and any other unexpected failure Рєњ
   `{"error": "footage_failed", "message": Рђд}`, with the real traceback kept in
   the server log (`log.exception`). The studio surfaces `message` to the user, so
   a future storage hiccup reads honestly instead of `internal_error`.

No AI added, no heuristic/template fallback Рђћ a write failure now surfaces an
honest error rather than a fabricated success (deterministic-engine boundary
respected; the store/ingest path is measurement + storage, not judgement).

> This closes the out-of-scope note QA-009 (batch 4) recorded Рђћ re-homing the
> media store's default DB + uploads under `DATA_DIR` Рђћ done the minimal way:
> only the *defaults* change (explicit-path callers and every other call site are
> untouched, since they already pass paths or go through the same singleton). The
> `_DEFAULT_DB` symbol QA-009's empty-state fallback referenced is replaced by the
> `_default_db_path()` accessor (caller + its test migrated 1:1, behaviour
> identical).

**Regression tests** (`tests/test_video_routes.py`):
`test_media_store_default_paths_follow_data_dir` (fails-before / passes-after Рђћ
before the fix `store.db_path` resolved to `Рђд/src/mediahub/data.db`, the exact
deploy bug; after, to `DATA_DIR/data.db`, and a written blob lands under
`DATA_DIR`) and `test_footage_upload_valid_mp4_does_not_500` (a valid small MP4 Рђћ
real H.264+AAC when FFmpeg is present Рђћ posted to `/api/video/footage` returns

## QA-011 [P2 quality] Рђћ Result time CLIPS / WRAPS / collides across studio archetypes

**Symptom.** In `/studio` with the default sample (Eira Hughes ┬и `2:08.41` ┬и
200M Freestyle ┬и Manchester Swimming Club), switching archetypes clipped or
wrapped the result time Рђћ the single most important fact on a swim card:
- **Broadcast Scorebug** Рђћ `2:08.41` clipped to `2:08.4` at the scorebug's
  right edge (final `1` cut).
- **Quote Led Recap** Рђћ `2:08.41` wrapped to `2:08.` / `41`, the second line
  overflowing the card's bottom edge.
- **Full Bleed Photo Lower Third** Рђћ the `NEW PB` kicker collided with the club
  lockup, and the event collided with the result chip.

Big Number Dominant and Magazine Cover rendered the same data perfectly Рђћ so it
looked per-archetype. An audit across all 29 archetypes at the studio preview
geometry found **16 of 29** clipping / wrapping / overflowing the result time.

**Investigation (real Chromium, measured DOM).** Rendered every archetype at the
studio **preview** geometry and at the **native** geometry and measured the
result element's box. At native (1080├Ќ1350 Рђћ the Download PNG) **all 29 fit on
one line**, no clip, no overflow. At the studio preview geometry **(540├Ќ675)**
the same 16 archetypes failed. So this is not the data, the autofit maths, or
the layouts Рђћ it is the geometry the preview is composed at.

**Root cause.** `web/design_editor.py` rendered the live preview at **half the
native CSS geometry** (`PREVIEW_SCALE = 0.5` shrank `StudioParams.size` to
540├Ќ675) "for a snappy, light payload". The v2 archetypes are authored with
**fixed-px furniture** (paddings, labels, the hanging quote glyph, the scorebug
cells) that does *not* scale with the canvas, while the autofit hero vars *do*.
At half geometry the fixed furniture is ~2├Ќ over-weighted relative to the card,
squeezing the result slot until the time clips, wraps and overflows, and
colliding fixed-position elements. The full-resolution download composed
correctly because its geometry was native Рђћ which is exactly why the symptom
looked archetype-specific and why "confirm in the Download PNG" sometimes
looked clean.

**Fix (smallest safe Рђћ fix the geometry, not the 29 correct layouts).** The
preview now **composes at the same native geometry as the download**, so every
archetype's furniture keeps its authored proportions and the time fits. The
"snappy, light payload" is preserved at the **raster**, not the geometry:
- `StudioParams.size` is always native (preview and download);
- the preview renders at a light **DPR-1** capture (`render_quality == "fast"`),
  the download at the default profile;
- the finished native render is downsampled to `PREVIEW_SCALE` for the response
  (`preview_raster_size`) by `downscale_png_bytes` Рђћ same render cost and same
  payload size as before, but a correct composition.

The preview is now a faithful, lighter copy of the Download PNG (the QA's stated
expectation Рђћ "the preview shows the actual rendered PNG scaled"). The
deterministic engine, the autofit maths, and all 29 layout templates are
untouched; the collisions and the no-photo "awkward empty stage" were the same
preview-scale artifact and resolve with the geometry.

**Files.** `src/mediahub/web/design_editor.py`, `src/mediahub/web/web.py`
(`api_studio_render` Рђћ pass the preview quality, downsample the response raster).

**Regression tests** (`tests/test_qa_011_studio_result_time_fit.py`):
`test_preview_composes_at_native_geometry_not_half_scale` (deterministic, always
runs Рђћ the preview must compose at native geometry, lightness only at the
raster), `test_downscale_png_is_safe_and_lossless_noop_when_not_needed`, and a
Chromium-gated structural audit
`test_result_time_fits_in_every_studio_archetype` that renders **all 29**
archetypes with a 7-char time and asserts the time sits inside the card on one
line with no horizontal clip and no bottom overflow (fails-before at half-scale:
16/29 clip Рђћ proven; passes-after: all 29 fit). The existing studio size
contract in `tests/test_design_editor_g1_27.py` was re-pinned to the corrected
behaviour (preview composes at native; the light raster lives in
`preview_raster_size`), and its render stubs widened to the real
`render_html_to_png(Рђд, *, quality=Рђд)` signature Рђћ no test weakened.

**Noted, out of scope (separate root cause).** The top-right meet-name vs
**PB-rosette** overlap is the G1.22 icon-overlay (`sprint_hooks/icon_overlay.py`)
placement against archetypes that carry top-right header text. It exists at
native (production downloads) independently of the preview-scale bug, lives in a
different subsystem with its own parity tests, and is left for a focused
follow-up rather than folded into this geometry fix as a drive-by.

## QA-013 [P0 accuracy/trust] Рђћ Disqualified (DQ) swims celebrated as PBs / "biggest time drop of meet"

**Symptom.** The Athlete Spotlight for Kenneth Powell
(`/spotlight/<run>/city_of_brighton_hove:Powell,Kenneth`) celebrated his 200m
Backstroke **3:29.20** with three fabricated "achievements": `#1 ELITE HUGE PB Рђћ
14.01s (6.3%)`, `#2 CONFIRMED PB: 3:29.20 (was 3:43.21, -14.01s)`, and `#3
BIGGEST TIME DROP OF MEET`. But that swim was **disqualified** Рђћ it has no valid
result, and it was his *only* 200 Backstroke swim. A club posting "new PB Рђћ
biggest drop of the meet!" for a DQ'd swim is factually false and reputational
poison. This is the accuracy floor and hits every product-wide surface that
reads moments (Meet Recap **and** Athlete Spotlight).

**Evidence (pdftotext -layout).** Event 207 "Open/Male 200 LC Meter
Backstroke", in both the 11-year-olds and the 10-15 combined listings:
`--- Kenneth Powell 11 City of Brighton & Hove 3:29.20 DQ` Рђћ the place column is
`---` and the result column is `DQ`. Same shape for Oscar Yu's DQ'd 200 IM
(`2:55.00 DQ`, Event 302).

**Root cause (deterministic interpreter Рђћ `interpreter/rows.py`).** The DSQ
markers themselves were already known to the parser (`_DSQ_TOKEN` / `_DSQ_VALUES`
= `DQ/DNS/DNF/DNC/NS/SCR/WD`), and the canonical bridge already maps a
non-numeric time to `finals_time_cs=None` / `dq=True` / `status="dq"`, which
**every** downstream detector (`pb.py`, `standout_history.py`,
`recognition_swim/achievements/official_pb.py`) and the report loop already
exclude. The break was upstream of all of that: when a printed (now-void) time
*co-occurred* with the marker (`Рђд 3:29.20 DQ`), the interpreter never emitted
the marker Рђћ it emitted a **valid** time and dropped the `DQ`:
- **Multi-record line parser** (`_split_into_records` / `_record_to_swim`, the
  path HY-TEK PDFs use): records terminate at a time token, so the line split at
  `3:29.20` (emitting a valid swim) and the trailing `DQ` became an empty,
  discarded record.
- **Schema/table path** (`_extract_swim_from_cells`): a separate `DQ` status
  cell was an unknown column and ignored, while the time cell `3:29.20` parsed
  as a valid result.

So the DQ swim reached the engine as `finals_time_cs=20920, dq=False`, and
because **3:29.20 beat his known online PB of 3:43.21**, it surfaced as a 14.01s
PB / huge-improvement / biggest-drop-of-meet. (Oscar Yu's DQ'd IM had no beating
prior PB, so it stayed hidden Рђћ confirming the filter must act at parse time,
independent of any PB comparison.)

**Fix.** Honour the DSQ marker that already exists in the source, in **both**
extraction paths, so a disqualified swim carries the marker as its "time"
(Рєњ `finals_time_cs=None` / `dq=True` at the bridge, Рєњ excluded by every detector):
- `_split_into_records`: defer closing a record at a real time token when the
  *next* token is a DSQ marker, so `Рђд 3:29.20 DQ` stays one record terminated by
  the marker.
- `_record_to_swim`: when the terminal token is a DSQ marker and the preceding
  body token is a real time, drop that struck-out time from the body (so it
  isn't mis-read as club/age data) Рђћ the swim's time becomes the marker.
- `_extract_swim_from_cells`: if any cell's first token is a DSQ marker, let the
  marker be the time (covers `DQ`, `DNS`, rule-coded `DQ 8.2`, Рђд).

No AI, no template/heuristic fallback Рђћ purely applying the parser's existing
DSQ knowledge at the point of extraction. Valid swims (including a valid swim in
a *different* event by the same swimmer) are untouched.

**Files.** `src/mediahub/interpreter/rows.py`.

**Regression tests** (`tests/test_dq_excluded_from_moments.py`, all fail-before /
pass-after): the DQ line parses to a swim whose time is the marker not `3:29.20`,
in both the line parser and the schema path; the bridge marks it
`dq=True`/`status="dq"`/`finals_time_cs=None`; and end-to-end Рђћ raw DQ line Рєњ
canonical Рєњ detection Рђћ it fires **zero** moments even with a prior PB of
3:43.21 that *would* have produced the fake 14.01s drop. A valid-swim control
proves the detector wiring is live, and a cross-event control (Oscar Yu: DQ'd
200 IM **plus** a valid 200 Back PB) proves DQ rows are dropped per-swim while a
valid swim in another event by the same swimmer still produces its real PB.
