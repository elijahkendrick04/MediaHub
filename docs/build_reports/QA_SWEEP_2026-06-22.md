## QA-001 [P1 accuracy] — Duplicate moment cards from overlapping age-band re-prints

**Symptom.** HY-TEK reprints the *same* swim under multiple overlapping
age-band sub-headers that share one Event number, so the interpreter surfaced
identical duplicate recognition cards (review list #1=#2 Raleigh, #3=#4 Carden),
inflating the queue to 416 cards and "254 PBs" vs 129 online-confirmed.

**Evidence (pdftotext -layout).**
- Birdy Raleigh 200 Free 2:19.80 under `Event 202 Female 10-13 …` (line 118)
  **and** `Event 202 Female 13 Year Olds …` (line 165).
- Izzy Carden 200 Free 2:17.00 under `Event 202 Female 16 & Over` (line 241)
  **and** `Event 202 Female 17 & Over` (line 262).

**Root cause.** The exact-reprint de-dup at the canonical seam
(`pipeline/interpreter_bridge.py::_canonical_result_identity`) keyed on
`age_band` **and** `place`. The age-band reprint mutates exactly those two
fields (the banner changes; the place is relative to each printed band), so the
key differed between printings and the reprints were never collapsed. The
existing de-dup only caught page-break reprints (identical banner).

**Fix.** Re-key the identity on the *physical swim*:
`(event number + swimmer + final time)`, keeping distance/stroke/course/gender
(to pin the event when no number is printed) and the round / A-B-final label
(so a genuine heat vs final is still preserved). The key now **excludes**
`age_band` and `place` — the two fields the reprint mutates. The event number is
parsed from the header (`_event_number`) and stored on the result's `extra` for
auditability. Collapsed reprints still emit the explainable
`duplicate_results_collapsed` info warning.

**Files.** `src/mediahub/pipeline/interpreter_bridge.py`.

**Regression tests.** `tests/test_interpreter_bridge_dedup.py`:
`test_age_band_reprint_of_same_event_collapses_to_one_swim`,
`test_distinct_swimmers_in_overlapping_bands_are_kept`,
`test_end_to_end_age_band_reprint_does_not_duplicate`.

## QA-002 [P2 trust/quality] — Meet title was the HY-TEK license/software banner

**Symptom.** Meet title showed
`Sussex County ASA- LC Champ - Organization License HY-TEK's MEET MANAGER 8.0 -
6:29 PM 15/02/2026 Page 1` (PDF line 1 = licensee + software + timestamp + page
banner), and it propagated into the generated "Weekend at a glance" headline.
The real meet name is PDF line 2:
`Sussex County Long Course Championships 26 - 14/02/2026 to 01/03/2026`.

**Root cause.** `interpreter/__init__.py::_extract_meet_metadata` took the first
non-trivial line as the meet name. On a HY-TEK results page that first line is
the licensee/software/timestamp/page banner, not the title.

**Fix.** Skip software/license/page **banner** lines (and event-header / result
rows) when choosing the title, picking the first plausible title line instead
(line 2 here), and strip a trailing date range. A degenerate header where the
name only survives merged into the banner recovers the prefix by stripping the
`Organization License` / `HY-TEK's MEET MANAGER …` / `Page N` / timestamp
boilerplate. Detection is structural (vendor + report-furniture words), so it
stays on the deterministic side of the engine boundary — no AI, no swim
vocabulary.

**Files.** `src/mediahub/interpreter/__init__.py`.

**Regression tests.** `tests/test_interpreter_corpus.py`:
`test_hytek_license_banner_is_not_the_meet_name`,
`test_hytek_banner_only_header_strips_boilerplate_to_recover_name`.

## QA-003 [P2 accuracy] — Phantom "<Club> NT" clubs in the club picker

**Symptom.** The configure club list contained phantom clubs
"City of Brighton & Hove NT", "Beacon SC NT", "Atlantis SC NT", etc.

**Evidence.** PDF line 282:
`Birdy Raleigh 13 City of Brighton & Hove  NT  2:24.51` — "NT" sits in the
seed-time column (No Time), not the team name.

**Root cause.** In the multi-record line parser the seed column sits between the
club and the finals time, so the club cell built from the tokens between the age
and the finals time absorbed the "NT" seed token
(`_normalise_club("City of Brighton & Hove NT")` → "… NT").

**Fix.** `interpreter/rows.py::_normalise_club` now strips a standalone trailing
No-Time seed marker ("NT" / "N.T."). Anchored on a preceding word boundary
(string start or whitespace) so a real club ending in those letters (… Kent,
… Trent) is untouched; a bare "NT" seed cell mis-mapped to the club column is
rejected entirely.

**Files.** `src/mediahub/interpreter/rows.py`.

**Regression tests.** `tests/test_interpreter_rows_normalisers.py`:
`test_no_time_seed_marker_is_stripped_from_club`,
`test_club_ending_in_nt_letters_is_untouched`,
`test_bare_no_time_marker_is_not_a_club`; end-to-end
`tests/test_interpreter_corpus.py::test_no_time_seed_does_not_create_phantom_clubs`.

## QA-004 [P2] — Design Studio (`/studio`) live preview stuck "Rendering…"

**Symptom.** The card preview never resolved to a clean card — it sat under a
persistent "Rendering…" layer (a semi-transparent, `backdrop-filter:blur`
overlay that read as garbled/"mirrored & doubled"), from a fresh page load and
across archetypes.

**Investigation.**
- (a) **The server PNG is correct, not mirrored/doubled.** Rendered the real
  studio path (`build_brief_from_params` → `render_brief`) at the preview size
  (540×675) *and* full (1080×1350) for both Big Number Dominant and Ribbon
  Banner — every output is a single, correctly-oriented card. The garble was the
  stuck overlay sitting over the card, not the pixels.
- (b) **The full-res "Download PNG" is unaffected** server-side (same clean
  render at native size); the overlay bug affected the on-page experience for
  both preview and download, never the downloaded file.

**Root cause.** The overlay is shown/hidden via its `hidden` attribute
(`showOverlay`/`hideOverlay`), but the scoped CSS set
`.mh-studio-overlay { display:flex }` — an author rule that overrides the UA
`[hidden]{display:none}`. With no `[hidden]` guard, `hideOverlay()`
(`overlay.hidden = true`) could never actually hide it, so "Rendering…" stayed
stuck over every render (confirmed in Chromium: computed `display` stays `flex`
even with `hidden` set).

**Fix.** Add `.mh-studio-overlay[hidden] { display:none; }` so the overlay can
be hidden. (`src/mediahub/web/design_editor.py`.)

**Regression tests** (`tests/test_design_editor_g1_27.py`):
`test_studio_overlay_is_hidable_via_its_hidden_attribute` (fails-before /
passes-after); plus a Chromium-gated contract test
`test_studio_render_is_a_single_unmirrored_card` (preview + full): exact
dimensions, top≠bottom (not doubled), real asymmetric content (not blank /
mirror-symmetric).

## QA-005 [P3] — Run progress bar non-monotonic (28%→10%→28%, steps 5↔3)

**Root cause.** The deployment runs `--workers 2` (gunicorn). Status polls
round-robin across both workers; the worker that isn't running the pipeline has
no in-memory log and falls back to the throttled, lagging DB `progress_log`, so
a single poll's `percent` / `log.length` can read lower than an earlier one.
`recap_progress` is monotonic per log — the non-monotonicity is the cross-worker
source mismatch, which only the client (the one consistent observer across
polls) can reconcile.

**Fix.** The run-progress page now holds a high-water mark for both the percent
and the step count / raw log, so reported progress never decreases. Verified in
Chromium: an input sequence `28→10→28→32%` / steps `5→3→5→6` clamps to
`[28,28,28,32]` / `[5,5,5,6]`. (`src/mediahub/web/web.py`.)

**Regression test** (`tests/test_run_status_dev_vs_customer.py`):
`test_progress_page_clamps_percent_and_steps_monotonically`.

## QA-006 [P2/P3] — Silent 403 on org finalise

**Root cause.** `POST /api/organisation/finalise` (the "Looks right — start
creating" cascade) sent only `Accept: application/json` — **no `Content-Type`**
and no CSRF token. The app's CSRF guard exempts writes whose content-type is
`application/json`; without that marker the same-origin POST was rejected 403,
so the derived palette was never persisted on finalise (the flow proceeded
anyway via the `<a href=/make>` fallback, masking the failure).

**Fix.** The cascade fetch now sends `Content-Type: application/json` (+ an empty
JSON body) — the documented CSRF-exempt marker, matching every other JSON write
in the app. Confirmed end-to-end: under enforced CSRF, the old shape → 403, the
fixed shape → 200 with the derived palette persisted. (`src/mediahub/web/web.py`.)

**Regression tests** (`tests/test_looks_right_button.py`):
`test_finalise_post_sets_json_content_type_to_dodge_csrf` (fails-before /
passes-after) and `test_finalise_csrf_contract_old_shape_blocked_json_shape_succeeds`
(end-to-end: old→403, fixed→200).

(covers `/sites` + `/documents`) and once in the season timeline. This fixes any
stale stored title (the displayed banner becomes `Sussex County ASA- LC Champ`)

heuristic guessing — pure boilerplate stripping, on the deterministic side of
the engine boundary.

## QA-009 [P2/P3] — Media Library shows "store wasn't readable" instead of a clean empty state

**Symptom.** An organisation that has never uploaded media (repro: "Swansea
University Swimming Team", `000 ASSETS`) opens `/media-library` and the
asset-list section renders the alarming
*"Couldn't load library assets — the store wasn't readable. Uploads above still
work; if this persists, ask your operator to check the data volume."* It
**persists across deploys** (re-checked on a healthy build — not transient).
**Expected:** the clean empty state ("No assets uploaded yet. Drop a photo above
to get started.") when the store is simply empty / not-yet-created for the org.

**Root cause.** The page (`media_library_page`, `web/web.py`) wrapped the media
read in a blanket `except Exception → store_failed = True`, so **every** read
failure — including the benign "no store on disk yet" case — rendered the
corrupt-store recovery message. The asset DB (`media_library/store.py`) is
created lazily on first **write**; its default path
(`src/mediahub/data.db`) lives in the *application* directory, **not** under
`DATA_DIR`. On a deployment where that directory is read-only (Render's code
layer), the first `get_store()` / `store.list()` for a fresh org can't create
the file and raises `OSError` / `sqlite3.OperationalError("unable to open
database file")` rather than returning `[]`. The route mislabelled that as
corruption. (A genuinely empty *writable* store already returns `[]` and showed
the empty state correctly — only the raising path was mishandled.)

**Fix.** Distinguish *"missing but fine"* from *"genuinely corrupt"* by the one
honest signal available: **is the DB file actually on disk?** On a read failure,
if the store's `db_path` (or the module default, when the store couldn't even be
constructed) does **not** exist → there is nothing wrong, just nothing uploaded
→ render the clean empty state. Only when the DB file **is** present but won't
read → keep `store_failed = True` and surface the "check the data volume"
recovery message. No new dependency, no engine touched, no AI/heuristic
fallback — a benign-empty store is reported honestly as empty, a corrupt one is
still reported honestly as corrupt. (`src/mediahub/web/web.py`.)

> Note (out of scope, recorded): the deeper smell is that the media store's
> default DB path is hardcoded to `src/mediahub/data.db` instead of being
> derived from `DATA_DIR`. Re-homing it under `DATA_DIR` would let uploads
> persist on read-only-code-dir deploys, but it's a data-location change across
> many call sites (uploads, deletes, DSR, video projects, org lifecycle) with
> migration implications — a separate, larger change, not folded into this
> empty-state display fix.

**Files.** `src/mediahub/web/web.py`.

**Regression tests** (`tests/test_media_library_empty_state.py`):
`test_missing_store_renders_empty_state_not_error` and
`test_store_construction_failure_with_no_db_renders_empty_state` (both
fail-before / pass-after — the empty state is shown, the error is not), plus the
guard `test_genuinely_corrupt_store_still_shows_recovery_message` (DB file
present but invalid still surfaces the recovery message — passes both ways).

# QA Sweep — 2026-06-22 (batch 5: QA-010)

A fifth live-site hand-off — a P1 in the Video Studio footage→reel flow. Same
rules: root cause, smallest safe fix, a regression test that fails before /
passes after, full suite green. (Storage-path + honest-error fix; no engine

## QA-010 [P1] — Video Studio footage upload returns HTTP 500 (footage→reel blocked at step 1)

**Symptom.** Opening `/video` and uploading a clean, valid 5-second 720×1280
portrait H.264 (yuv420p) + AAC MP4 (271 KB, well under the 50 MB-raised cap)
returned `POST /api/video/footage` **500**; the studio showed
`Upload failed: internal_error` and `No footage yet`. A volunteer could not make
a reel at all. Reproduced twice.

**Root cause.** The media-library store (`media_library/store.py`) — which the
footage-ingest path writes through (`video/ingest.py::ingest_footage_stream` →
`store.store_blob_stream` / `store.save`) — hardcoded its blob + `data.db`
defaults to the **package dir** (`Path(__file__).resolve().parents[1]`, i.e.
`/app/src/mediahub/uploads_v4/…` and `/app/src/mediahub/data.db`) instead of
deriving them from `DATA_DIR`. On the deploy the code tree is **root-owned** and
the runtime user is **uid 10001** (`Dockerfile` chowns only `/app/runs_v4`,
`/app/uploads_v4`, `/app/.cache`, `/app/data` to `mediahub`; everything else
under `/app` stays root). Writing a footage blob there raised **`PermissionError`**
(a subclass of `OSError`), which the route did not catch — only `ValueError` was
handled — so it fell through to the global 500 handler that emits
`{"error": "internal_error"}`. The sibling Video-Studio store
(`video/projects.py::_data_dir`) and every other store (workflow schedule,
autonomy, review-comments, audio upload) already key off `DATA_DIR`; the media
library was the lone outlier, also violating the CLAUDE.md **DATA_DIR rule**
("all storage paths derived from `DATA_DIR`; never hardcode"). Any data that
*did* land in the package dir was also lost on redeploy.

**Fix.**
1. **Root cause** — `media_library/store.py` now resolves its default DB +
   uploads dir from `DATA_DIR` at construction time (`_data_dir()` /
   `_default_db_path()` / `_default_uploads_dir()`), mirroring `video/projects.py`
   exactly, with the same package-dir fallback only when `DATA_DIR` is unset.
   Footage/photo blobs now land on the writable persistent disk
   (`/var/mediahub/uploads_v4/media_library`, `/var/mediahub/data.db`). Explicit
   `db_path`/`uploads_dir` args (tests, callers) are unchanged.
2. **Honest error (no more `internal_error`)** — `api_video_footage_upload` now
   catches `OSError` → `{"error": "storage_failed", "message": …}` (an honest,
   specific "couldn't be saved — storage full or unavailable" in the style of the
   meet-file/audio upload errors) and any other unexpected failure →
   `{"error": "footage_failed", "message": …}`, with the real traceback kept in
   the server log (`log.exception`). The studio surfaces `message` to the user, so
   a future storage hiccup reads honestly instead of `internal_error`.

No AI added, no heuristic/template fallback — a write failure now surfaces an
honest error rather than a fabricated success (deterministic-engine boundary
respected; the store/ingest path is measurement + storage, not judgement).

> This closes the out-of-scope note QA-009 (batch 4) recorded — re-homing the
> media store's default DB + uploads under `DATA_DIR` — done the minimal way:
> only the *defaults* change (explicit-path callers and every other call site are
> untouched, since they already pass paths or go through the same singleton). The
> `_DEFAULT_DB` symbol QA-009's empty-state fallback referenced is replaced by the
> `_default_db_path()` accessor (caller + its test migrated 1:1, behaviour
> identical).

**Regression tests** (`tests/test_video_routes.py`):
`test_media_store_default_paths_follow_data_dir` (fails-before / passes-after —
before the fix `store.db_path` resolved to `…/src/mediahub/data.db`, the exact
deploy bug; after, to `DATA_DIR/data.db`, and a written blob lands under
`DATA_DIR`) and `test_footage_upload_valid_mp4_does_not_500` (a valid small MP4 —
real H.264+AAC when FFmpeg is present — posted to `/api/video/footage` returns
non-500 with `ok:true`, never `internal_error`).

# QA Sweep — 2026-06-22 (batch 6: QA-014)

## QA-014 [P0 accuracy/trust] — Parser read the SEED (entry) time as the time SWAM

**Symptom.** Every displayed time, PB, time-drop, barrier/"first-time" and
medal-time claim was built on the wrong time — and many were **inverted**: a
swimmer who swam *slower* than their seed was reported as setting a big PB.

Objective proof (Sussex County Championships 2026, run `72c223c97b54`, no
column-order assumption — it uses the printed FINA points, computed from the
actual swum time):

- **Event 207**, 200 LC Backstroke, Oscar Yu placed **6th** with **227 FINA
  pts** (behind Jake Evans' 3:00.18 / 239). FINA points descend with place, and
  only the *second* time column (`2:41.44 < 2:50.45 < 2:54.01 < 2:58.63 <
  3:00.18 < 3:03.47 < 3:18.84`) is monotonic with place — so the **swum time is
  the second column (Finals Time)**; the first is the Seed. Yu swam **3:03.47**,
  not his 2:55.80 seed. Yet the Athlete Spotlight showed
  *"#1 BIG PB 7.67s — 2:55.80 (was 3:03.47)"* — it used the seed as the result
  and the real swum time as the "previous best", celebrating a **7.67s
  regression as a 7.67s improvement**.
- **Event 202**, 200 LC Freestyle: Birdy Raleigh and Effie Maxted have an
  **identical first-column 2:19.80** but placed 3rd vs 4th (500 vs 475 pts) —
  identical swum times can't place differently, so the first column is the
  **seed**. Raleigh swam **2:22.28** (the second column), i.e. **over 2:20** —
  yet MediaHub fabricated a *"first time under 2:20"* headline.

**Root cause (deterministic results parser — fixed deterministically, no AI).**
In the HY-TEK PDF/text layout the swum result is the **rightmost time column
before the FINA points** — labelled *"Finals Time"*, or *"Prelim Time"* on a
prelim-only sheet. The file uses three header layouts (*Seed | Finals*,
*Prelim | Finals*, *Seed | Prelim*); in **all** of them the result is the
rightmost time and *"Seed Time"* is **never** the result. The collapsed-line
parser (`interpreter/rows.py`, the path that actually fired on this PDF) closed
each record at the **first** time token, so it took the **Seed** as the swum
time and discarded the real result. The seed flowed through the bridge into
`RaceResult.finals_time_cs`; the PB / barrier / improvement detectors compare
`finals_time_cs` against the verified prior-best history (they deliberately
never use the seed), so a seed mistaken for the result inverted every claim.

**Fix.** Select the result by column **label** (priority Finals > Prelim; never
Seed), not by fixed position, and keep the seed separate:

1. **Header-driven schema path** — `data/ontology/column_headers.json` gains a
   `seed_time` header family (`Seed Time`/`Seed`/`Entry Time`/`Entry`), listed
   before `time` so an exact/partial match resolves a seed column to `seed_time`
   rather than the generic `time`; `time` gains the `Finals Time`/`Prelim Time`
   labels. The seed column is therefore typed `seed_time` and can never be
   chosen as the result; among the remaining `time` columns the rightmost
   (Finals when both Prelim+Finals are present) wins, unchanged.
2. **Collapsed-line parser** — a record now closes at the **last** time token of
   a consecutive run of time tokens (keeping a seed+result pair together instead
   of slicing the seed off as the time). `_record_to_swim` peels that trailing
   run, takes the **last** non-reaction time as the swum result and the **first**
   as the seed. A trailing **reaction** column (`0.63`, time-shaped but a start
   reaction) is excluded from result/seed selection. The two-records-per-line
   layout still splits correctly (the token after a record's time there is the
   next record's place number, not a time).
3. `InterpretedSwim` gains a `seed_time` field so raw extraction stays distinct
   from the canonical swum time (per the "separate raw from canonical" data
   rule). The canonical bridge is unchanged (`seed_time_cs` stays as-is), so no
   downstream behaviour shifts beyond the corrected result time.

No AI, no heuristic guessing — the selection is the printed column label plus the
fixed HY-TEK structure (result = rightmost time before the points). Stays on the
deterministic side of the engine boundary.

**Result after fix.** Oscar Yu 200 Back = **3:03.47** (a regression vs his
2:55.80 seed) → **no PB/improvement card**. Birdy Raleigh 200 Free = **2:22.28**
→ not sub-2:20, no "first time under 2:20". Every card's time is the
Finals/Prelim swum time, never the Seed.

**Regression tests** (`tests/test_interpreter_seed_vs_result_time.py`, 18 cases,
fail-before / pass-after — before the fix Oscar Yu parsed as `2:55.80`):
the line parser and the header-driven schema path both return Finals (`3:03.47`)
as `time` and `2:55.80` as `seed_time` across all three header layouts; the
`Prelim | Finals` layout prefers Finals; a trailing reaction column is never the
result; the two-records-per-line split is preserved; the full
`assign_rows_to_events` path over the Sussex Event 207 field yields Finals times
monotonic with finishing order; and `PBConfirmedDetector` fires **no** PB for
the correct (slower) 3:03.47 while a genuine faster swim still fires (positive
control) and the old seed-as-result fabricates the 7.67s PB (inversion locked
out).
